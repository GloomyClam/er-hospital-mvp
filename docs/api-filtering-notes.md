# 2단계 필터링 검토 메모

현재 사용 중인 API는 `getEmrrmRltmUsefulSckbdInfoInqire`입니다.

이 오퍼레이션은 응급실 실시간 가용 병상 정보를 가져오는 용도라서, 환자 상태 입력값을 병원 조건과 직접 매칭하기보다는 "현재 수용 가능한 병상/시설 여유가 있는지"를 판단하는 데 더 적합합니다.

## 확인용 로그

서버에서 `/api/hospitals`가 호출되면 `src/services/emergencyApi.js`가 공공 API 응답의 `item` 원본 배열을 서버 콘솔에 출력합니다.

로그 라벨:

```text
[emergencyApi] getEmrrmRltmUsefulSckbdInfoInqire raw item fields:
```

브라우저 콘솔이 아니라 `npm start`를 실행한 터미널에서 확인합니다.

## 현재 응답에서 필터링에 쓸 수 있는 값

프로젝트에서 이미 사용 중인 값:

| 필드 | 현재 이름 | 의미 | 필터링 가능성 |
| --- | --- | --- | --- |
| `hpid` | `id` | 기관 ID | 병원 식별, 상세 API 연결 키 |
| `dutyName` | `name` | 병원명 | 이름 검색 |
| `dutyTel3` | `phone` | 응급실 연락처 | 연락처 표시 |
| `hvec` | `generalBeds` | 응급실 가용 병상 | 응급실 수용 가능 여부 |
| `hvoc` | `surgeryBeds` | 수술실 가용 여부/수 | 수술 필요 가능성의 약한 조건 |
| `hvgc` | `totalBeds` | 입원실 가용 병상 | 입원 가능성의 약한 조건 |
| `hvicc` | `icuBeds` | 중환자실 가용 병상 | 중증 환자 수용 가능성의 약한 조건 |
| `hvidate` | `updatedAt` | 데이터 갱신 시각 | 오래된 정보 제외/경고 |

콘솔 로그에서 추가로 확인할 값:

| 후보 필드 | 확인 이유 |
| --- | --- |
| `hv*` 계열 전체 | CT, MRI, 분만실, 격리병상, 소아/음압 등 세부 가용 자원이 포함될 수 있음 |
| 주소/좌표 계열 | 거리순 정렬이나 현재 위치 기반 필터링 가능 여부 확인 |
| 진료과/전문센터 계열 | 환자 증상과 직접 매칭 가능한 정보가 현재 응답에 있는지 확인 |

## 현재 API만으로 가능한 필터링

현재 API 하나만으로 비교적 바로 가능한 것:

- 응급실 병상이 1개 이상 있는 병원만 보기
- 중환자실 병상이 1개 이상 있는 병원만 보기
- 수술실 병상이 1개 이상 있는 병원만 보기
- 입원실 병상이 1개 이상 있는 병원만 보기
- 갱신 시간이 너무 오래된 병원 제외 또는 경고 표시
- 병원명 검색

단, 병상 수가 있다고 해서 해당 환자의 증상/질환을 실제로 진료할 수 있다는 뜻은 아닙니다.

## 추가 API가 필요할 수 있는 필터링

추가 조회가 필요할 가능성이 큰 것:

- 증상별 추천: 흉통, 복통, 외상, 소아, 산부인과, 뇌졸중 의심 등
- 진료과별 필터링: 내과, 외과, 정형외과, 신경외과, 소아청소년과 등
- 장비/센터 기반 필터링: CT, MRI, 심혈관센터, 뇌혈관센터, 외상센터 등
- 위치 기반 추천: 사용자 현재 위치에서 가까운 순서
- 병원 상세 정보: 주소, 대표전화, 운영시간, 기관 등급, 응급의료기관 종류

이 경우 `hpid`를 기준으로 기관 상세 정보 조회, 위치 정보 조회, 진료과/기관 정보 조회 같은 추가 오퍼레이션을 붙이는 방식이 필요할 수 있습니다.

## 현재 판단

2단계에서 "환자 상태 입력 후 조건 필터링"을 단순히 병상 여유 기준으로만 시작한다면 현재 API 하나로 MVP 구현은 가능합니다.

하지만 환자 상태를 실제 진료 가능 여부와 연결하려면 현재 실시간 병상 API만으로는 부족할 가능성이 높습니다. 다음 단계 전에 콘솔에 찍힌 `item` 전체 필드를 확인해서, 증상/진료과/장비/주소에 해당하는 값이 응답에 포함되는지 먼저 확인하는 것이 좋습니다.
